# FPS 서버 아웃라인 작업 기록

날짜: 2026-03-15  
위치: `TheBoat/Server`

## 요청 요약

`Server/Server` 안에 있던 기존 Boost.Asio 예제를, 단순 소켓 수신 샘플이 아니라 PUBG 같은 FPS 게임 서버의 시작점이 될 수 있는 구조로 바꾸는 것이 목적이었다.

## 변경 전 상태

- `BoatServer`는 TCP 클라이언트를 받아서 수신한 바이트를 출력하는 정도만 처리하고 있었다.
- 세션 관리, 플레이어 상태, 매치메이킹, 매치 상태, 서버 틱 같은 개념이 없었다.
- 프로젝트는 이미 Boost.Asio를 참조하고 있었고, 진입점은 단순한 `main` 형태였다.

## 이번 작업에서 한 판단

### 1. 완성품처럼 보이게 만드는 대신, 확장 가능한 아웃라인으로 정리

한 번에 FPS 서버의 어려운 문제를 다 해결하는 척하는 건 의미가 없다고 판단했다. 대신 이후 기능을 붙이기 쉬운 뼈대를 먼저 만드는 방향으로 정리했다.

- 연결/세션 계층
- 명령 처리 계층
- 플레이어 상태 컨테이너
- 매치 상태 컨테이너
- 고정 틱 진입 지점

이렇게 해두면 나중에 실제 게임 로직을 붙일 때 기존 예제를 통째로 버리지 않아도 된다.

### 2. `ClientSession` 도입

판단 이유:

- 기존 코드는 accept와 read 로직이 `BoatServer`에 바로 섞여 있었다.
- FPS 서버는 연결별 송신 큐, 연결 종료 처리, 클라이언트별 식별자가 필요하다.

변경 내용:

- TCP 클라이언트 1개를 `ClientSession` 1개로 대응시켰다.
- 읽기는 `async_read_until` 기반의 줄 단위 처리로 바꿨다.
- 쓰기는 세션별 큐를 두고 순차 전송하게 만들었다.
- 연결 종료는 `BoatServer` 쪽으로 다시 전달되게 만들었다.

### 3. 서버가 플레이어와 매치 상태를 권위적으로 관리하도록 변경

판단 이유:

- FPS 게임은 클라이언트를 진실의 원천으로 두면 안 된다.
- 아웃라인 단계라도 플레이어 위치와 매치 소속은 서버가 관리하는 형태가 맞다.

변경 내용:

- 플레이어 id, 이름, 위치, 체력, 매치 소속 등을 담는 `FPlayerState`를 추가했다.
- 매치 phase와 참가자 목록을 담는 `FMatchState`를 추가했다.
- 서버가 플레이어/매치를 id 기준 컨테이너로 보관하도록 만들었다.

### 4. 매치메이킹 큐와 매치 생성 흐름 추가

판단 이유:

- 배틀/FPS 서버는 “클라이언트 연결됨” 수준에서 멈추면 의미가 없다.
- 최소한 플레이어가 매치에 들어가는 흐름은 보여야 이후 확장이 가능하다.

변경 내용:

- `QUEUE` 명령으로 플레이어를 매치메이킹 큐에 넣도록 했다.
- 일정 수 이상이 모이면 매치를 생성하게 했다.
- 플레이어에게 `MatchId`를 부여하도록 했다.
- 서버가 `MATCH_CREATED` 메시지를 브로드캐스트하도록 했다.

### 5. 고정 Tick 루프 추가

판단 이유:

- FPS 서버는 보통 고정된 간격으로 권위적 시뮬레이션을 돈다.
- 실제 전투 시뮬레이션이 아직 없더라도, 그 로직이 들어갈 자리는 미리 있어야 한다.

변경 내용:

- `boost::asio::steady_timer` 기반으로 20Hz tick을 추가했다.
- `RunTick()`에서 현재는 매치 phase 전환과 snapshot 브로드캐스트를 수행한다.
- 이후 이동 검증, 피격 판정, 안전 구역, 루팅, 차량, 동기화 규칙 등을 붙일 확장 지점으로 잡았다.

### 6. 프로토콜은 당장은 단순하게 유지

판단 이유:

- 현재 단계에서는 텍스트 명령만으로도 흐름 검증이 가능하다.
- 바이너리 패킷 설계는 서버 책임 구조가 더 명확해진 뒤 들어가는 편이 낫다.

현재 명령:

- `HELLO <name>`
- `QUEUE`
- `MOVE <x> <y> <z>`
- `FIRE <payload>`
- `PING`

현재 응답:

- `WELCOME`
- `HELLO_ACK`
- `QUEUE_ACK`
- `MATCH_CREATED`
- `MATCH_STATE`
- `PLAYER_MOVED`
- `PLAYER_FIRED`
- `SNAPSHOT`

## 일부러 아직 넣지 않은 것

- 바이너리 패킷 직렬화
- UDP 전송 또는 TCP/UDP 혼합 네트워킹
- 랙 보정
- 리와인드 기반 히트 판정
- interest management
- 인벤토리, 루팅, 자기장, 차량, 리스폰 규칙
- 인증 및 재접속 처리
- DB/스토리지 연동

이런 것들은 규모가 큰 시스템이라, 전송 방식과 시뮬레이션 계약이 먼저 안정된 뒤 넣는 것이 맞다고 판단했다.

## 검증 상태

MSBuild로 빌드 확인을 시도했다. 다만 컴파일 검증까지는 가지 못했고, 프로젝트가 참조하는 NuGet Boost 패키지가 빠져 있어서 그 단계에서 막혔다.

- 누락 파일: `..\packages\boost.1.86.0\build\boost.targets`

즉 현재 막힌 지점은 서버 아웃라인 로직 자체보다 패키지 복원 상태다.

## 다음 권장 작업

1. Boost NuGet 패키지를 복원해서 프로젝트가 실제로 빌드되게 만든다.
2. 서버 코드를 세션, 프로토콜, 매치메이킹, 매치 시뮬레이션 파일로 분리한다.
3. 텍스트 프로토콜을 메시지 id와 시퀀스 번호가 있는 바이너리 패킷 구조로 바꾼다.
4. 이동/전투 트래픽을 당분간 TCP로 둘지, 빠르게 UDP로 옮길지 초기에 결정한다.
5. 클라이언트 예측을 붙이기 전에 서버 권위 이동 검증부터 먼저 넣는다.
